- 日付: 2026-06-15
- 環境: Windows 11 / Blender 5.0.1（addon reachable・token あり）
- schema_hash: `af6b81c144578bdbf9eeffb96dbb54e03dd793b57eff861baca2e791990cd52a`
- 検証者: エージェント（CLI 経由でのみ Blender を操作・**ビューポートは不可視**）

---

## 0. TL;DR

- ツールの基盤は問題なく動作した（診断→参照→補正→検証→復元の一連が CLI で通る）。
- ただし依頼「支柱に対してやや傾いた対象を真上に向ける」は **`straighten` 単体では完遂できなかった**。
- 主因は2つ:
  1. **`pca` が符号で上下反転する**（重心ベースの符号決定が、ベースが重いスキャン物体で破綻）。
  2. **正しい補正を確認・指定する手段が無い**（dry-run / 目視 / undo / 基準指定がいずれも未提供）。
- 結果、正しい回転はエージェントが **手計算で算出して `transform` 経由で適用**する迂回を要した。
- この「迂回が必要だった」事実が、後述フィードバック（特に #1〜#5）の必要性を直接裏付ける。

## 1. 対象と前提

`object-info` の結果:

| 項目 | 値 |
|---|---|
| name | `UMesh_kei_syuusei_ecoshi1` |
| type / 規模 | MESH / 249,999 頂点・500,091 ポリゴン（スキャンメッシュ） |
| location | `[0, 0, 0]` |
| scale | `[0.01, 0.01, 0.01]` |
| **rotation_euler_deg** | **`[90, 0, 0]`**（ぴったり・インポート軸変換と思われる） |
| dimensions | `[8.571729, 14.4055, 9.766603]` |

**重要な観察:** シーン内オブジェクトはこの **1個のみ**。依頼にある「支柱」は**別オブジェクトとして存在しない**（同一スキャンメッシュの一部）。
さらに回転が厳密に `[90,0,0]` という綺麗な値のため、「やや傾いている」分の傾きは **オブジェクト transform ではなくメッシュ形状そのものに焼き込まれている**。

→ この時点で「object 回転だけを触る方法（reset/world-align）」では傾きを補正できないことが確定する。

---

## 2. `straighten` 4メソッドの適合性評価

| method | 判定 | 理由 |
|---|---|---|
| `reset` | ✗ | 全回転を identity 化 → 意図した 90° インポート補正まで消える。形状の傾きには無関係。 |
| `world-align` | ✗ | object transform しか見ない。今回は既に綺麗（local +Y が +Z を向く）→ 形状の傾きには **no-op**。 |
| `pca` | △ | 唯一の「形状ベース」で発想は正しいが、**符号で上下反転**＋不要な**ヨー回転**が混入し、そのまま bake は不可。 |
| `floor` | ✗ | 平行移動のみ・回転しない。 |

## 3. `pca` の具体的な失敗内容（再現データ）

`straighten --method pca`（**bake なし＝可逆**）で観測:

```json
{
  "method": "pca",
  "eigenvalues": [1.33316578, 3.43830452, 5.1607063],
  "principal_world": [-0.194687, 0.160503, -0.967644],
  "principal_world_after": [0.0, -0.0, 1.0],
  "rotation_euler_deg": [-99.4179, 11.2265, 100.067]
}
```

- 主成分軸（最大分散方向）は **`[-0.195, 0.160, -0.968]`** ＝ ほぼ真下向き。垂直からのずれは **約 14.6°**（＝「やや傾いている」を定量化できた点は良い）。
- ところが符号決定（`_principal_axis`: `原点→重心` 方向を + に揃える）が、**ベースが重い＝重心が下にある**スキャン物体で **下向きを「principal +」と確定**。
- その結果 `+Z` へ合わせる回転が **約 165°** になり、**オブジェクトが上下反転**する（適用後 euler `[-99.4, 11.2, 100.1]`）。
- つまり `--bake-rotation` を付けて実行していたら、**モデルが逆さまに焼き込まれていた**。

> 根本原因: PCA の符号曖昧性の解決を「重心オフセット」に依存している点。ベース/土台が質量的に偏るスキャン物体では、しばしば「下」を正と誤判定する。

---

## 4. 完遂できなかった理由（まとめ）

1. **傾きがメッシュ側に焼き込まれている** → object 回転を触るだけの `reset` / `world-align` では補正不能。
2. **形状ベースの `pca` は符号で反転** → そのままでは使えない。
3. **正しい補正を「確認」する手段が無い**:
   - dry-run が無く、`pca` の計画値を見るには **一度適用してから戻す**しかない。
   - **ビュー（スクリーンショット/レンダー）が無く**、傾きの向きも結果も目視確認できない。
   - **undo コマンドが無い**（addon 内部に `push_undo` はあるが CLI 未公開）。可逆性は「直前 transform を自力再構築」で担保するしかない。
4. **「支柱を基準に」整列する手段が無い** → 全体 PCA しか選べず、支柱だけを基準にできない。

## 5. エージェントが取った迂回策（参考）

`straighten` では決まらないため、**反転しない最小回転**を手計算して `transform`（bake なし・可逆）で適用した。

1. 上向き主成分 `p_up = -principal = [0.195, -0.160, 0.968]` を `+Z` へ移す最小回転 delta を算出（≈ **14.6°**）。
2. `R_new = delta @ R_cur` を構築し、euler に変換。
3. **euler 規約の較正が必要だった**: ツール出力（`pca` 適用後 euler）と突き合わせ、Blender の `XYZ` euler が **`Rz @ Ry @ Rx`** 規約であることを実測で確定（L1 差 = 0.0）。
4. 算出値 **`rotation_euler_deg = [80.5821, -11.2264, 0.9277]`** を適用。読み戻しが計算値と完全一致。

**検証（幾何から独立に再測定）:**

| | 補正前 | 補正後 |
|---|---|---|
| 主成分軸（world） | `[-0.195, 0.160, -0.968]` | **`[0, 0, -1]`**（水平成分ゼロ） |
| 垂直からの傾き | 約 **14.6°** | **0°** |

> 注: この迂回は「最大分散軸＝意図する上方向」という前提に依存する。本来はツール側が基準指定で解決すべき。

## 6. フィードバック（優先順）

### #1【最優先】ビュー取得（スクリーンショット / レンダー）
- 例: `bli screenshot --out x.png` / `bli render --camera <name> --out x.png`
- 「見た目をまっすぐに」系タスクは、**現状エージェントが完全に目隠し**。傾きの向きも結果の妥当性も検証できない。スキル化の前提として最重要。

### #2 dry-run / プレビュー（特に mutating 系）
- 例: `straighten --dry-run` → 適用せず「計画回転・主成分・up からの角度」を返す。
- 今回は計画値を見るために「適用→復元」を要した。dry-run があれば安全に「確認してから実行」できる。

### #3 undo コマンドの公開
- 例: `bli undo`（addon の `push_undo` を露出）。
- 可逆性を「直前 transform の自力再構築」に頼らせない。試行錯誤の安全性が大幅に上がる。

### #4 基準指定の整列（支柱問題の本丸）
- 参照オブジェクトや **選択した一部ジオメトリ（＝支柱）** を基準に整列。
- 案: `straighten --reference <obj>` / 選択面のみで PCA / 明示角度 `straighten --method angle --rotate-axis ... --degrees ...`。
- 全体 PCA では「支柱を基準」が表現できない。

### #5 PCA の符号・ヨー制御 ＋ 角度レポート
- `--up-hint=current`（**現在の up に最も近い符号を選ぶ**）/ `--keep-up` を追加。今回の上下反転を防げる。
- up 軸のみ補正し、**不要なヨー（方位回転）を加えない**オプション。
- 結果に **`tilt_from_up_deg`（up からの角度）** を含める → 呼び出し側が妥当性を即チェックできる。

### #6 非破壊の傾き計測
- 読み取り専用で主成分軸・垂直からの角度を返す（`object-info` 拡張 or `bli measure --tilt`）。
- 今回は計測のために mutating 系（`straighten`）を呼ぶ必要があった。

### #7 細かい点
- `object-info` のフラグが `--target` ではなく **`--targets`**（複数形）。直感に反するのでエイリアス検討。
- Windows で標準出力が **文字化け**（CP932）。`PYTHONUTF8=1` 等で UTF-8 を強制しないと日本語サマリーが読めない。出力を UTF-8 固定にしてほしい。
- `dimensions`（回転不変なオブジェクト固有サイズ）と `bbox.size`（world AABB）の違いが紛らわしい。ドキュメント明記を推奨。

## 7. 最小の改善提案（このケースを一発で通すなら）

`straighten` に **`--up-hint=current`（符号を現在 up 寄りに）** と **`--dry-run`** を足すだけで、今回の迂回（手計算・euler 規約較正）は不要になり、

```
bli straighten --targets <obj> --method pca --up-hint current --dry-run
# → tilt_from_up_deg ≈ 14.6, planned rotation ≈ [80.58, -11.23, 0.93] を確認
bli straighten --targets <obj> --method pca --up-hint current
```

の2手で安全に決まるはずである。さらに #4（基準指定）が入れば「支柱基準」の本来意図にも応えられる。

## 付録: 確認できた良い点

- 非破壊（bake なし）操作は **完全に可逆**で、安全に試行できる設計（`_apply_world_rotation` が loc/scale を保持）。
- `straighten` の結果に `eigenvalues` / `principal_world` / `principal_world_after` を含めており、**外部から検算できた**（今回の解析が成立したのはこのおかげ）。
- `verified` / `fingerprint` による結果検証フィールドがあり、復元の同一性確認に使えた。
- エラーメッセージが具体的（`--axis は world-align のときのみ有効` 等）で、原因特定が速い。
